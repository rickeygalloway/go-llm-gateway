Overview

What Is the LLM Gateway?

The Go LLM Gateway is a reverse proxy and routing layer for large language models. Instead of your application talking directly to a specific AI provider, it talks to the gateway — which then intelligently decides which backend provider to use, handles failures, and translates between APIs.

The gateway exposes a fully OpenAI-compatible REST API, meaning any code that already works with OpenAI (LangChain, LiteLLM integrations, custom clients) can point to the gateway with a single URL change and immediately gain multi-provider routing.

🔀

Intelligent Routing

Define fallback chains per model. If Ollama is down, traffic automatically shifts to OpenAI or Anthropic without any application-level code changes.

🔌

The Problem

Why Does This Exist?

Modern AI-powered applications face a common set of infrastructure problems that get worse as you scale. The LLM Gateway solves them at the platform layer, so every team using AI models in your organization benefits automatically.

❌ Without a Gateway

Every service hardcodes a provider SDK. Switching providers (e.g., moving from GPT-4 to Claude for cost) requires touching every codebase. Outages at one provider take down everything. Cost visibility is scattered across teams.


# 5 teams, 5 different implementations
team-a: openai.ChatCompletion.create(...)
team-b: anthropic.messages.create(...)
team-c: requests.post("ollama:11434/...")
team-d: openai.ChatCompletion.create(...)
team-e: # uses different API key, no fallback

✅ With a Gateway

All teams use one URL. Provider strategy, fallback logic, API key rotation, cost controls, and observability are all centralized. A provider outage triggers automatic failover — applications don't notice.


# All teams, one interface
ALL_TEAMS → http://llm-gateway/v1

# Gateway handles everything:
if ollama_healthy:   → route to ollama (free)
elif anthropic_ok:  → route to anthropic
else:               → route to openai

🔒 Vendor Lock-in Eliminated

Your application code never imports anthropic or openai — only standard HTTP. Swap providers in a config file, not in code across 10 repos.

💰 Cost Optimization

Route simple queries to cheap local models (Ollama + Llama 3.2) first. Fall back to expensive frontier models only when necessary. Save 80%+ on inference costs for routine tasks.

🛡️ Resilience by Default

Provider outages, rate limits (429s), and 5xx errors are automatically retried on the next available provider. Your application SLA is no longer tied to a single vendor's uptime.

Getting Started

Quickstart

Get the gateway running locally in under 2 minutes. No credentials required for the Ollama-only setup.

1. Clone and Build

# Clone the repo
git clone https://github.com/go-llm-gateway/go-llm-gateway
cd go-llm-gateway

# Build the binary (auto-detects Go installation)
./build.sh

# Run all 31 unit tests
./build.sh test

2. Configure Providers

# config.yaml — minimal example
server:
  addr: ":8080"

providers:
  # Free local inference — tried first
  - type: "ollama"
    base_url: "http://localhost:11434"
    priority: 0
    models: ["llama3.2:3b", "mistral:7b"]

  # OpenAI fallback — used if Ollama fails
  - type: "openai"
    api_key: "sk-..."
    priority: 10

  # Anthropic fallback
  - type: "anthropic"
    api_key: "sk-ant-..."
    priority: 20

3. Start the Full Stack (with Ollama)

# Starts Ollama, Postgres, Redis, Kafka in Docker
docker compose up -d

# Start the gateway
./build.sh run

4. Send Your First Request

# Non-streaming completion
curl http://localhost:8080/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "llama3.2:3b",
    "messages": [
      {"role": "system", "content": "You are a helpful assistant."},
      {"role": "user", "content": "What is the capital of France?"}
    ]
  }'

# Streaming (SSE)
curl -N http://localhost:8080/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "llama3.2:3b",
    "messages": [{"role": "user", "content": "Tell me a story"}],
    "stream": true
  }'

# List all available models across all providers
curl http://localhost:8080/v1/models

# Check provider health
curl http://localhost:8080/readyz

Using with Python (OpenAI SDK)

from openai import OpenAI

# Point to the gateway — no other changes needed
client = OpenAI(
    base_url="http://localhost:8080/v1",
    api_key="not-needed",  # Gateway manages provider keys
)

response = client.chat.completions.create(
    model="llama3.2:3b",
    messages=[{"role": "user", "content": "Hello!"}]
)
print(response.choices[0].message.content)

Using with LangChain

from langchain_openai import ChatOpenAI

# Drop-in replacement — gateway handles routing
llm = ChatOpenAI(
    openai_api_base="http://localhost:8080/v1",
    openai_api_key="not-needed",
    model_name="llama3.2:3b",
)

result = llm.invoke("Summarize AI infrastructure in one paragraph.")

Key Design Decisions

Canonical Wire Format

All providers translate to and from a single canonical type defined in pkg/openai/types.go. Adding a new provider means implementing one translation layer — the router, handler, and streaming logic are untouched.

type Provider interface {
  Name() string
  Chat(ctx, req) (resp, error)
  ChatStream(ctx, req) (<-chan Chunk, <-chan error)
  ListModels(ctx) ([]Model, error)
  Health(ctx) error
}

Retryability Classification

The router makes intelligent fallback decisions based on error type. Not every error should trigger a retry — a 404 means the model doesn't exist on that provider; retrying wastes time and money.

func IsRetryable(err error) bool {
  // Retry: timeout, 429, 5xx
  // Fail fast: 400, 404
  pe, ok := err.(*ProviderError)
  if !ok { return true } // network err
  return pe.StatusCode == 429 ||
         pe.StatusCode >= 500
}

Anthropic Protocol Translation

Anthropic's API differs significantly from OpenAI's: system messages go in a separate top-level field, not in the messages array. The adapter handles this transparently — callers always use standard OpenAI format.

// OpenAI format (what callers send)
messages: [
  {role: "system", content: "..."},
  {role: "user",   content: "..."}
]

// Anthropic format (what we send)
system: "...",
messages: [{role: "user", content: "..."}]

SSE Streaming

Streaming uses Go channels for backpressure-safe communication between providers and the HTTP handler. Each provider streams chunks asynchronously; the handler writes data: {...} SSE frames as they arrive.

chunkCh, errCh := router.RouteStream(ctx, req)

for {
  select {
  case chunk := <-chunkCh:
    w.Write(sseFrame(chunk))
    w.Flush()
  case err := <-errCh:
    return err
  }
}

API Reference

Endpoints

The gateway exposes a fully OpenAI-compatible API. Any client that works with OpenAI's API works with the gateway.

POST /v1/chat/completions Chat completions with fallback routing

Send a chat completion request. Set "stream": true to receive SSE chunks.

// Request body
{
  "model": "llama3.2:3b",        // Any model available across providers
  "messages": [
    {"role": "system", "content": "You are helpful."},
    {"role": "user",   "content": "Hello!"}
  ],
  "temperature": 0.7,          // Optional
  "max_tokens": 256,           // Optional
  "stream": false              // Set true for SSE streaming
}

// Response
{
  "id": "chatcmpl-abc123",
  "object": "chat.completion",
  "model": "llama3.2:3b",
  "choices": [{
    "index": 0,
    "message": {"role": "assistant", "content": "Hi! How can I help?"},
    "finish_reason": "stop"
  }],
  "usage": {"prompt_tokens": 12, "completion_tokens": 6, "total_tokens": 18}
}
GET /v1/models List all models across all healthy providers 
// Response — aggregated and deduplicated across all providers
{
  "object": "list",
  "data": [
    {"id": "llama3.2:3b",     "owned_by": "ollama"},
    {"id": "gpt-4o",           "owned_by": "openai"},
    {"id": "claude-sonnet-4-6", "owned_by": "anthropic"}
  ]
}
GET /healthz Liveness probe — is the gateway process alive? 
{"status": "ok", "time": "2025-03-18T12:00:00Z"}
GET /readyz Readiness probe — are providers reachable?

Returns 200 if at least one provider is healthy, 503 if all are down.

{
  "status": "ready",
  "providers": [
    {"provider": "ollama",    "healthy": true},
    {"provider": "openai",    "healthy": true},

}

Scalability

How It Scales

The gateway is designed from the ground up for horizontal scaling. It is stateless by design — no in-memory state that needs sharing between instances. Kubernetes native with full health probe support.

Horizontal Scaling

The gateway is fully stateless. Deploy 10 replicas behind a load balancer and every instance is identical. No leader election, no shared cache required for the core routing path.

kubectl scale deployment llm-gateway \
  --replicas=10

Step 2: Redis Rate Limiting

The next step adds Redis-backed rate limiting per API key and per model. This allows fine-grained control — different teams get different request quotas — all enforced at the gateway layer.

Step 3: OpenTelemetry + Prometheus

Distributed tracing with OpenTelemetry lets you trace a request from client all the way to the provider and back. Prometheus metrics expose per-provider latency, token counts, and error rates for Grafana dashboards.

llm_gateway_request_duration_seconds
llm_gateway_tokens_total{provider,model}
llm_gateway_provider_errors_total{provider,code}

Step 5: pgvector Semantic Cache

Cache identical or semantically similar requests using pgvector. A question phrased 10 different ways can return the same cached answer. Reduces inference costs by 40-60% for applications with repeated queries.

# 1. Embed query → vector
# 2. Query pgvector by cosine similarity
# 3. Cache hit: return instantly (0ms)
# 4. Cache miss: route to provider

Step 4: Kafka Audit Trail

Emit every request and response to a Kafka topic for async processing. Enables cost attribution, compliance auditing, training data collection, and abuse detection — all without adding latency to the request path.

# kafka topic: llm-gateway.requests
{
  "team": "ml-platform",
  "model": "gpt-4o",
  "tokens": 1247,
  "cost_usd": 0.0037
}

Step 6: Kubernetes Operator

A custom K8s operator lets platform teams define provider configurations as Kubernetes CRDs. Developers submit a LLMProvider resource; the operator reconciles it into the gateway config without any restarts.

apiVersion: gateway.io/v1
kind: LLMProvider
spec:
  type: vllm
  base_url: http://vllm-svc:8000/v1
  priority: 5

Roadmap

Where It's Going

The gateway is structured as 6 progressive steps, each adding a production capability. Step 1 is complete and production-ready.

1

Multi-Provider Router + Full Scaffold ✓ Complete

OpenAI-compatible API, fallback chain, 4 provider adapters (Ollama, OpenAI, Anthropic, vLLM), SSE streaming, health probes, Dockerfile, Helm chart, 31 unit tests. Zero external dependencies for tests.

2

Rate Limiting (Redis-backed)

Per-API-key and per-model rate limits backed by Redis. Token bucket or sliding window algorithm. Distributed across all gateway replicas. Returns standard 429 responses with Retry-After headers.

3

OpenTelemetry Tracing + Prometheus Metrics

Distributed tracing with OTLP export. Prometheus metrics for request duration, token counts, provider errors, and cache hit rates. Pre-built Grafana dashboard included.

4

Kafka Audit Trail

Async event emission per request for cost attribution, compliance, training data capture, and abuse detection. Non-blocking Kafka producer so it adds zero latency to the request path.

5

pgvector Semantic Cache + Metadata Store

Embed requests and store in PostgreSQL with pgvector. Cache hits for semantically similar queries (cosine similarity threshold configurable). Request metadata store for analytics and debugging.

6

Kubernetes Operator (controller-runtime)

Custom CRD: LLMProvider. Operators reconcile provider configs into the running gateway without restarts. GitOps-friendly: merge a PR to add a new provider to the cluster.
